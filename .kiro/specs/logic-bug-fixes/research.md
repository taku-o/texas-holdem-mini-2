## リサーチログ

### BUG-G7: raiseAmountの負値発生条件

- **コンテキスト**: `applyAction`のRaise処理で`raiseAmount`が負になるケースの特定
- **調査結果**:
  - 現在の計算: `Math.min(p.chips, Math.max(minRaise, amount) - p.currentBet)`
  - `Math.max(minRaise, amount)`が`p.currentBet`より小さい場合に負になる
  - 例: `p.currentBet = 40`, `minRaise = 40`, `amount = 30` → `Math.max(40, 30) - 40 = 0`（境界）
  - 例: `p.currentBet = 50`, `minRaise = 40`, `amount = 30` → `Math.max(40, 30) - 50 = -10`（負値）
  - 負値の場合`Math.min(p.chips, -10)`で`chips`が負でなければ`-10`が選ばれ、`p.chips -= -10`でチップが増加する
- **影響**: `Math.max(0, ...)`で`raiseAmount`をラップすることで解決。既存の計算ロジック構造は維持

### BUG-E1: JavaScriptのSet順序保証

- **コンテキスト**: `findStraight`でのSet挿入順依存の調査
- **調査結果**:
  - ECMAScript 2015+ではSetは挿入順を保持する（仕様保証）
  - 現在は`cards.sort()`後にSetに挿入するため降順が保持される
  - ただし`flushCards`はスート別に集めたサブセットであり、元の`cards`配列のソート順に暗黙依存
  - 明示的ソートを追加することで、入力順序に関係なく正しい動作を保証
- **影響**: `uniqueVals.sort((a, b) => b - a)`の1行追加で解決

### BUG-G5: getNextActivePlayerの-1返却と呼び出し元への影響

- **コンテキスト**: 全員非アクティブ時の挙動変更が呼び出し元に与える影響
- **調査結果**:
  - 呼び出し元1: `useGameEngine.ts` L141 `advancePhase`内 → `firstToAct`として使用。showdownフェーズでは`activePlayerIndex: -1`が設定されるため、-1返却は既存の挙動と整合
  - 呼び出し元2: `useGameEngine.ts` L189 `handleAction`内 → `nextActive`として使用。ここに到達する前に`isRoundOver`で全員フォールドが検出される
  - 結論: -1返却は既存コードフローで安全。追加のガード不要
- **影響**: ループカウンタによる上限チェック追加で解決。戻り値-1は既存フローとの整合性あり

### BUG-G4: advancePhaseのテスト戦略

- **コンテキスト**: `advancePhase`がReact Hook内のローカル関数であるためテスト困難
- **調査結果**:
  - 現在の`advancePhase`はフェーズ遷移、コミュニティカード配布、プレイヤーリセット、デッキ操作を担当
  - フェーズ遷移のコミュニティカード配布ロジックを純粋関数として`gameLogic.ts`に抽出することでテスト可能になる
  - 既にフェーズ3で`getNextActivePlayer`, `isRoundOver`, `applyAction`等を`gameLogic.ts`に抽出したパターンがあり、同じアプローチが適用可能
- **影響**: `dealCommunityCards`関数の抽出が必要。`useGameEngine.ts`は抽出した関数を呼び出すように変更

## 設計判断

### 判断: BUG-G4のバーンカード実装アプローチ

- **コンテキスト**: `advancePhase`内のバーンカード追加をどのようにテストするか
- **検討した選択肢**:
  1. `advancePhase`内に直接`newDeck.pop()`を追加し、E2Eテストのみで検証
  2. フェーズ遷移のカード配布ロジックを`gameLogic.ts`の純粋関数`dealCommunityCards`として抽出し、単体テストで検証
- **選択したアプローチ**: オプション2 — `dealCommunityCards`純粋関数の抽出
- **理由**: フェーズ3で確立した抽出パターンに従い一貫性を保つ。単体テストで正確なデッキ残枚数の検証が可能。バーンカードの有無を独立してテスト可能
- **トレードオフ**: 変更ファイルが増えるが、テスト品質が向上する
- **フォローアップ**: 抽出後に既存E2Eテストがパスすることを確認

## リスクと緩和策

- BUG-G5の-1返却による呼び出し元への影響 — 調査の結果、既存フローで安全であることを確認済み
- `dealCommunityCards`抽出によるリグレッション — 既存E2Eテスト + 単体テストで検証
- バーンカード追加によるデッキ枚数不足 — 52枚デッキで最大消費枚数: ホール(10) + バーン(3) + コミュニティ(5) = 18枚。十分な余裕あり
