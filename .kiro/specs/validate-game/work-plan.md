# 作業計画書: ゲーム検証・修正

## 基本方針

**UI・デザインの保護を最優先とする。**
すべての作業ステップの前後でUIが壊れていないことを自動テストで検証する。
**ユーザーによる目視確認・手動確認は一切行わない。** すべての検証はテストコードによる数値ベースの自動検証で行う。

### UI検証の方法

1. **Playwrightスクリーンショット比較**: 各画面状態のスクリーンショットをベースライン画像として保存し、以降のテストでピクセル単位で差分比較する（`toHaveScreenshot()`）
2. **要素の数値検証**: 要素数、テキスト内容、表示/非表示、CSS計算値（色、サイズ、位置）を数値として取得しアサーション
3. **BoundingBox検証**: 要素の座標・幅・高さを取得し、レイアウト崩れがないことを数値で検証

## フェーズ 0: UI保護の仕組み構築

> **目的**: 以降の全作業でUIリグレッションを自動検出できる基盤を作る

### タスク 0-1: Playwright環境構築
- `@playwright/test` のインストール
- `playwright.config.ts` の作成（devサーバー自動起動設定含む）
- スクリーンショット比較の閾値設定（`maxDiffPixelRatio` 等）
- `package.json` にスクリプト追加（`test:e2e`、`test:e2e:update` ベースライン更新用）

### タスク 0-2: 初期画面のE2Eテスト + スクリーンショットベースライン
- タイトル「Texas Hold'em」のテキスト取得 → 文字列一致検証
- 「Start Game」ボタンの存在 → 要素数 = 1
- ボタンの BoundingBox 取得 → 幅 > 0、高さ > 0、画面内に収まっていること
- **初期画面のスクリーンショットベースライン保存**（`toHaveScreenshot('idle-screen.png')`）

### タスク 0-3: ゲーム画面レイアウトのE2Eテスト + スクリーンショットベースライン
- ゲーム開始後のプレイヤー情報要素数 = 5
- ポット金額要素のテキストが `$` を含み数値を含む
- Fold / Check(Call) / Raise ボタン要素数の検証
- ロールバッジ（D / SB / BB）の表示テキスト検証
- 各プレイヤーの BoundingBox 取得 → 座標が画面内に収まっている
- テーブル要素の BoundingBox → 幅・高さが最低値以上
- **ゲーム画面のスクリーンショットベースライン保存**（`toHaveScreenshot('game-screen.png')`）

### タスク 0-4: カード表示のE2Eテスト
- ヒューマンプレイヤーのカード → ランク文字（2〜A）が表示されている要素数 = 2
- CPUプレイヤーのカード → ランク文字が表示されていない（裏面）ことを検証
- カード要素の BoundingBox → 幅・高さが期待範囲内（例: 幅 50-70px, 高さ 75-100px）

### タスク 0-5: プレイヤー操作のE2Eテスト
- 自ターン時: コントロール要素の `opacity` CSS値 ≠ 0.5（pointer-events: none でない）
- 非自ターン時: コントロール要素の `opacity` CSS値 = 0.5 または `pointer-events` = `none`
- Foldクリック後: プレイヤーのアクションバッジテキスト = "fold"
- Check/Callボタンのテキスト: `callAmount > 0` なら "Call $数値" / それ以外なら "Check"
- Raiseボタン: 金額テキストに数値が含まれる

### タスク 0-6: ゲームフロー統合E2Eテスト
- ゲーム開始 → コミュニティカード枚数遷移の数値検証
  - pre-flop: コミュニティカード表示枚数 = 0
  - flop: コミュニティカード表示枚数 = 3
  - turn: コミュニティカード表示枚数 = 4
  - river: コミュニティカード表示枚数 = 5
- アクションログ要素数 ≥ 1
- ショーダウン後のポット金額 = $0（分配済み）

**チェックポイント**: `npm run test:e2e` 全テストパス。スクリーンショットベースライン確立。

## フェーズ 3: ゲームロジックの抽出とテスト

> **目的**: useGameEngine内のロジックを純粋関数として抽出し、テスト可能にする
> **UIへの影響**: 関数の抽出のみ。振る舞いは変えない。

### タスク 3-1: ロジック関数の抽出
`src/utils/gameLogic.ts` に以下を抽出:
- `getNextActivePlayer(currentIndex, players)`
- `isRoundOver(players, currentBet)`
- `calculateBlinds(players, dealerIndex)`
- `applyAction(state, action, amount)` → 新しいstateを返す純粋関数
- `determineWinner(players, communityCards)`

`useGameEngine.ts` は抽出した関数を呼び出すように変更（振る舞いは変えない）。

### タスク 3-2: 抽出した関数のテスト (`gameLogic.test.ts`)
- `getNextActivePlayer`: 戻り値のインデックス数値が正しい / フォールド済み・チップ0のインデックスを返さない
- `isRoundOver`: `true`/`false` の戻り値検証
- `calculateBlinds`: 返されるインデックス数値（dealer, sb, bb, utg）の検証
- `applyAction`:
  - Fold: 戻り値 `player.action` = `'fold'`
  - Call: 戻り値 `player.chips` = 元値 - コール額, `pot` = 元値 + コール額
  - Raise: 戻り値 `player.currentBet` ≥ `currentBet * 2`
- `determineWinner`: 勝者の `player.id` が期待値と一致

**チェックポイント**: `npm run test` パス + `npm run test:e2e` パス（スクリーンショット差分なし）。
リファクタリングによるUI変化がないことをスクリーンショット比較で自動検証。

## 各フェーズ共通の確認フロー

```
作業実施
  ↓
npm run test（単体テスト）→ パス件数・失敗件数を数値で報告
  ↓ 失敗 0 件
npm run test:e2e（Playwright E2Eテスト + スクリーンショット比較）→ パス件数・失敗件数を数値で報告
  ↓ 失敗 0 件
[UI変更があるタスク] スクリーンショットベースラインを更新（npm run test:e2e:update）
  ↓
次のタスクへ
```

UI変更のないタスクでは、スクリーンショット差分が 0 であることが自動検証される。
UI変更のあるタスクでは、変更後にベースラインを更新し、以降はその新しいベースラインで検証する。
