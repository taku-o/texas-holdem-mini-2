# テスト計画書: ゲームロジック・UI検証

## 概要

テキサスホールデムのゲームロジックとUIの正しさを検証するためのテスト計画。

---

## Part 1: ゲームロジックのテスト（単体テスト）

### テストフレームワーク

- **Vitest**: Viteネイティブのテストランナー。Viteの設定をそのまま利用可能。
- **@testing-library/react** + **@testing-library/jest-dom**: Reactコンポーネントのテスト用。

### テスト対象と項目

#### 1.1 デッキ (`src/utils/deck.ts`)

| テストケース | 検証内容 |
|------------|----------|
| デッキ生成で52枚のカードが作られる | `createDeck()` の戻り値が52枚 |
| デッキに重複カードがない | 全カードがユニーク |
| 全スート・全ランクが含まれる | 4スート × 13ランク |
| シャッフルで順番が変わる | `shuffleDeck()` の戻り値が元と異なる順序 |
| シャッフルで枚数が保持される | シャッフル後も52枚 |

#### 1.2 ハンド評価 (`src/utils/evaluator.ts`)

| テストケース | 検証内容 |
|------------|----------|
| ロイヤルフラッシュの判定 | A-K-Q-J-10同スート |
| ストレートフラッシュの判定 | 5連続同スート |
| フォーカードの判定 | 同ランク4枚 |
| フルハウスの判定 | 3枚 + 2枚 |
| フラッシュの判定 | 同スート5枚以上 |
| ストレートの判定 | 5連続 |
| ホイールストレート（A-2-3-4-5） | Aceがローとして機能 |
| スリーカードの判定 | 同ランク3枚 |
| ツーペアの判定 | 2枚ペア × 2 |
| ワンペアの判定 | 2枚ペア × 1 |
| ハイカードの判定 | 役なし |
| スコアの大小関係 | ロイヤルフラッシュ > ストレートフラッシュ > ... > ハイカード |
| 同じ役でのキッカー比較 | ワンペア同士でキッカーの高い方が勝つ |
| 7枚から最良の5枚選択 | ホール2枚 + コミュニティ5枚から最強ハンドを選ぶ |
| 空カードへの対応 | カードが0枚の場合にエラーにならない |

#### 1.3 ゲームエンジン ロジック部分

ゲームエンジンのロジックは現在`useGameEngine`フックに集約されているため、テスト可能にするにはロジック部分の抽出が必要。

**方針A: ロジック関数の抽出（推奨）**

以下の関数を純粋関数として`src/utils/gameLogic.ts`に抽出し、単体テスト可能にする:

| 抽出対象関数 | テストケース |
|------------|------------|
| `getNextActivePlayer(currentIndex, players)` | 次のアクティブプレイヤーを正しく取得 / フォールド済みをスキップ / チップ0をスキップ |
| `isRoundOver(players, currentBet)` | 全員行動済みでラウンド終了 / 未行動者がいれば未終了 / ベット未一致で未終了 |
| `calculateBlinds(players, dealerIndex)` | SB/BBの正しいポジション配置 / 非アクティブプレイヤーのスキップ |
| `applyAction(state, action, amount)` | Fold: プレイヤーがフォールド状態になる / Call: 正しい金額を支払う / Raise: ベット額が正しく反映 |
| `determineWinner(players, communityCards)` | 最強ハンドのプレイヤーが勝者になる |

**方針B: フック全体のテスト**

`renderHook`を使って`useGameEngine`をテスト。状態遷移をシミュレートする。方針Aより複雑だが、統合テストとしてより網羅的。

| テストケース | 検証内容 |
|------------|----------|
| ゲーム開始 | 5人のプレイヤーが初期チップ1000で生成される |
| ブラインド投入 | SBが10、BBが20を支払い、ポットが30になる |
| フェーズ遷移 | pre-flop → flop(3枚) → turn(1枚) → river(1枚) → showdown |
| 全員フォールド | 1人以外フォールドで即座にポット獲得 |
| ショーダウン | 最強ハンドのプレイヤーがポットを獲得 |
| 破産プレイヤー | チップ0のプレイヤーが次のハンドで非アクティブになる |
| ゲームオーバー | アクティブプレイヤーが1人になったら終了 |

## Part 3: 導入手順

### 3.1 Vitest導入

```bash
npm install -D vitest @testing-library/react @testing-library/jest-dom jsdom
```

`vite.config.ts`にテスト設定を追加:
```typescript
/// <reference types="vitest/config" />
export default defineConfig({
  // ... 既存設定
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/test/setup.ts'],
  },
});
```

`package.json`にスクリプト追加:
```json
{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest"
  }
}
```

テストファイル配置:
```
src/
  utils/
    __tests__/
      deck.test.ts
      evaluator.test.ts
      gameLogic.test.ts    ← 方針A採用時
  hooks/
    __tests__/
      useGameEngine.test.ts ← 方針B採用時
  test/
    setup.ts
```

### 3.2 Playwright導入

```bash
npm install -D @playwright/test
npx playwright install
```

テストファイル配置:
```
e2e/
  game-start.spec.ts
  game-layout.spec.ts
  card-display.spec.ts
  player-controls.spec.ts
  game-flow.spec.ts
```

`package.json`にスクリプト追加:
```json
{
  "scripts": {
    "test:e2e": "playwright test"
  }
}
```

## Part 4: 推奨実装順序

1. **Vitest環境構築** — テストフレームワークのセットアップ
2. **deck.test.ts** — 最もシンプルな純粋関数のテスト
3. **evaluator.test.ts** — ハンド評価のテスト（全役パターン）
4. **ゲームロジック抽出 + テスト** — 方針Aでロジックを抽出してテスト
5. **Playwright環境構築** — E2Eテストのセットアップ
6. **E2Eテスト実装** — 画面表示・操作のテスト

## Part 5: 方針の選択（ユーザー確認事項）

以下の点についてユーザーの判断が必要:

1. **ゲームロジックのテスト方針**: 方針A（ロジック抽出 + 単体テスト）と方針B（フック全体テスト）のどちらを採用するか。推奨は方針A。
2. **テストの優先度**: 全テスト項目を実装するか、重要度の高い項目のみから始めるか。
3. **調査報告書の問題点への対応**: テスト作成と同時に問題修正も行うか、テスト作成のみに集中するか。
