## パラメータの解決

- **feature名**（$ARGUMENTS）: 対象のfeature名

### パラメータが指定されていない場合の処理

$ARGUMENTS（feature名）が空の場合：
1. Claude Codeのメモリ（memory配下のファイル群）を確認し、直前まで作業していたfeature名を特定してください。
2. 特定できない場合は、「直前まで作業していたfeatureを特定できませんでした。feature名を指定して再実行してください。」とエラーメッセージを表示して処理を中止してください。

解決したパラメータをユーザーに表示してから作業を開始してください（例：「feature: xxx を対象にします」）。

以降、解決したfeature名を {feature} として使用します。

## 精査観点

### 1. requirements.md の精査

以下の観点でチェックしてください：

**EARS形式の準拠**
- 受け入れ基準がEARSパターン（When/While/If/Where/The [system] shall）に従っているか
- トリガーキーワードは英語のまま、変数部分のみターゲット言語で記述されているか
- "shall" は必須要件、"should" は推奨事項に適切に使い分けられているか

**要件ID**
- 全ての要件見出しに数値IDが含まれているか（"Requirement A" のようなアルファベットIDは不可）
- IDが連番になっているか

**要件の品質**
- 受け入れ基準が「WHAT（何をするか）」の記述であり、「HOW（どう実装するか）」の詳細が含まれていないか
- 各受け入れ基準が単一の振る舞いを記述しているか（複数の振る舞いが混在していないか）
- テスト可能・検証可能な記述になっているか
- 曖昧な言語（「素早く」「適切に」「多数の」等）が使われていないか

**ドキュメント構造**
- Introduction セクションが存在するか
- 各要件に Objective（As a [role], I want [capability], so that [benefit]）が記述されているか
- Acceptance Criteria セクションが各要件に存在するか


### 2. design.md の精査

以下の観点でチェックしてください：

**要件トレーサビリティ**
- requirements.md の全要件IDがdesign.mdでカバーされているか
- Requirements Traceability セクションまたは各コンポーネントのRequirements欄に、要件IDが正確に記載されているか

**設計書の構造**
- Overview（Purpose/Users/Impact）が記述されているか
- Goals / Non-Goals が明確に定義されているか
- Architecture Pattern & Boundary Map が説明されているか（複雑な機能ではMermaid図を含むか）
- Components and Interfaces が定義されているか
- Error Handling Strategy が記述されているか
- Testing Strategy が記述されているか

**アーキテクチャ整合性**
- steeringドキュメント（tech.md / structure.md）に記載された既存アーキテクチャパターンと整合しているか
- 依存方向が適切か（レイヤー境界を越えていないか）
- 命名規則がプロジェクト標準に沿っているか

**設計の品質**
- 実装詳細（コードの具体的な書き方）が設計書に混入していないか
- ドキュメントが1000行を大幅に超えていないか（超えている場合は複雑すぎるサイン）
- Non-Goals に将来拡張・最適化・フォールバック等の不要な項目が含まれていないか


### 3. tasks.md の精査

以下の観点でチェックしてください：

**フォーマット準拠**
- チェックボックス形式（`- [ ] 1. メジャータスク` / `- [ ] 1.1 サブタスク`）になっているか
- ネストが最大2レベルに収まっているか（1.1.1 のような深いネストが存在しないか）
- メジャータスクの番号が連番（1, 2, 3...）になっているか
- サブタスクの番号がメジャータスクごとにリセット（1.1, 1.2, 2.1, 2.2...）されているか

**要件マッピング**
- 各サブタスクの詳細末尾に `_Requirements: X.X_` 形式で要件IDが記載されているか
- requirements.md の全要件IDがtasks.mdでカバーされているか
- 要件IDが数値形式（X.X形式）で記載されているか（テキストラベルが混在していないか）

**タスク記述の品質**
- タスクが自然言語で機能・成果を記述しているか
- ファイルパス・メソッド名・型定義・クラス名等の実装詳細が記述されていないか（これらはdesign.mdに属する）
- デプロイ・ドキュメント作成・ユーザーテスト・マーケティング等のコード以外のタスクが含まれていないか

**タスクの設計**
- 並列実行可能なタスクに `(P)` マーカーが付いているか
- 統合タスクが最後に配置されているか
- タスクが前のアウトプットを積み上げる形になっているか（孤立したコードが生まれないか）


### 4. 専門家観点レビュー（ソフトウェアエンジニアリング）

フォーマットや整合性だけでなく、ソフトウェアエンジニアリングの専門家として内容の妥当性を評価してください。

**requirements.md の内容レビュー**
- 要件に漏れがないか（主要なユースケース、エッジケース、異常系が考慮されているか）
- 要件間に矛盾や競合が存在しないか
- 技術的に実現可能な要件か（現在のプロジェクト構成・技術スタックを踏まえて）
- セキュリティに関する要件が必要なのに欠落していないか（認証・認可・入力検証・データ保護等）
- パフォーマンスやスケーラビリティに関する考慮が必要な箇所で欠落していないか
- エラー発生時やデータ不整合時のユーザー体験が考慮されているか
- 要件のスコープが適切か（過大すぎて1フィーチャーに収まらない、または過小で意味のある価値を提供しない）

**design.md の内容レビュー**
- 選択されたアーキテクチャパターンが要件に対して妥当か（過剰設計・過少設計でないか）
- コンポーネント間の責務分離が適切か（単一責任原則に沿っているか）
- コンポーネント間のインターフェース定義が明確で、暗黙の依存がないか
- データフローに不整合や循環依存がないか
- エラーハンドリング戦略が現実的か（エラーの伝播経路が明確か、リカバリ方法が定義されているか）
- セキュリティ上の懸念がある設計判断がないか（安全でないデータの受け渡し、権限チェックの欠落等）
- 状態管理の設計が複雑すぎないか（競合状態やデータ不整合のリスクがないか）
- テスト戦略が要件のリスクレベルに見合っているか（重要な機能に十分なテストカバレッジが計画されているか）
- 既存コードベースへの影響範囲が特定・考慮されているか

**tasks.md の内容レビュー**
- タスクの粒度が適切か（大きすぎて作業が不明確、または小さすぎて管理コストが過大でないか）
- タスクの依存関係と実行順序が論理的に正しいか（先に必要な基盤が後に配置されていないか）
- リスクの高いタスク（新技術の導入、複雑なロジック、外部連携等）が早期に配置されているか
- 統合・結合時に発生しうる問題が考慮されたタスク分割になっているか
- タスク完了時の検証方法が暗黙的にでも明確か（何をもって「動く」と判断するか）


### 5. クロスドキュメント整合性

**requirements.md ↔ design.md**
- requirements.md に記載された全要件がdesign.mdで設計されているか
- design.md に登場する要件IDがrequirements.mdに実在するか（幽霊IDが存在しないか）

**requirements.md ↔ tasks.md**
- requirements.md に記載された全要件がtasks.mdでカバーされているか
- tasks.md に記載された要件IDがrequirements.mdに実在するか

**design.md ↔ tasks.md**
- design.md で設計されたコンポーネントがtasks.mdに実装タスクとして存在するか
- tasks.md のタスクがdesign.mdのアーキテクチャ境界を尊重しているか
