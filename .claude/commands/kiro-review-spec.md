仕様駆動開発の仕様書セット（要件定義書・設計書・タスクリスト）を精査して、問題がないか確認してください。

## パラメータの解決

- **feature名**（$ARGUMENTS）: 対象のfeature名

### パラメータが指定されていない場合の処理

$ARGUMENTS（feature名）が空の場合：
1. Claude Codeのメモリ（memory配下のファイル群）を確認し、直前まで作業していたfeature名を特定してください。
2. 特定できない場合は、「直前まで作業していたfeatureを特定できませんでした。feature名を指定して再実行してください。」とエラーメッセージを表示して処理を中止してください。

解決したパラメータをユーザーに表示してから作業を開始してください（例：「feature: xxx を対象にします」）。

以降、解決したfeature名を {feature} として使用します。

対象フィーチャー: {feature}

## 実行手順

### Step 1: 対象ファイルの事前読み込み

以下のファイルを自分で読み込んでください（存在しないファイルはその旨を報告して処理を中止すること）：
- `.kiro/specs/{feature}/requirements.md`
- `.kiro/specs/{feature}/design.md`
- `.kiro/specs/{feature}/tasks.md`
- `.kiro/specs/{feature}/spec.json`

参照ルール・テンプレートも読み込んでください：
- `.kiro/settings/rules/ears-format.md`
- `.kiro/settings/rules/tasks-generation.md`
- `.kiro/settings/rules/design-review.md`
- `.kiro/settings/templates/specs/requirements.md`
- `.kiro/settings/templates/specs/design.md`
- `.kiro/settings/templates/specs/tasks.md`

また、steeringコンテキストも全て読み込んでください：
- `.kiro/steering/` 配下の全ファイル（product.md, tech.md, structure.md, カスタムファイル含む）

### Step 2: 並列レビューの実行

Agent toolを使って、以下の6つのエージェントを**全て同時に（1つのメッセージで）**起動してください。
各エージェントのプロンプトには、Step 1で読み込んだファイルの内容を全て含めてください。
エージェントはファイルを読むだけで、編集は行いません。

#### Agent 4: 非対応項目・曖昧コミットメント検出（subagent_type: "Explore"）

description: "Detect ambiguous commitments"

プロンプトに含める内容：

```
あなたは仕様書レビューの専門家です。以下の3つのドキュメント全体を通して、項目としては挙がっているが実際には対応しない（または対応が保証されない）記載がないかチェックしてください。

【検出対象となる表現パターン】
以下のような表現が含まれる項目を全て抽出してください：
- 「オプション」「オプショナル」「optional」
- 「推奨」「recommended」「should（義務ではない文脈で）」
- 「場合によっては対応」「必要に応じて」「状況に応じて」
- 「可能であれば」「余裕があれば」「時間が許せば」
- 「将来的に」「今後の課題」「後日対応」「Phase 2」「次フェーズ」
- 「検討する」「検討中」「要検討」
- 「Nice to have」「あると望ましい」「理想的には」
- 「TBD」「未定」「要確認」
- その他、対応の確約を避ける同様の表現

- 上記の表現を含む項目を全て列挙し、それぞれについて以下を判定すること：

#### Agent 6: 既存ドキュメント影響範囲チェック（subagent_type: "Explore"）

description: "Check existing docs impact"

プロンプトに含める内容：
- requirements.md の全文

指示内容：

```
あなたは仕様書レビューの専門家です。今回の仕様書セット（requirements.md / design.md / tasks.md）に基づく修正が、プロジェクト既存の説明系ドキュメントと齟齬を生じさせる場合、その更新タスクがtasks.mdから漏れていないかを検出してください。

【調査対象となる既存ドキュメント】
プロジェクトルートおよび配下を自分で探索し、以下を列挙して読み込んでください：
- プロジェクトルート直下の `README.md` および `README.*.md`（英語版・日本語版など多言語対応のバリエーションを含む）
- `docs/` ディレクトリ配下の全マークダウンファイル（再帰的に探索すること）
- その他、ユーザー向け・開発者向けの説明ドキュメントとして機能している `*.md`（例: `CONTRIBUTING.md`, `ARCHITECTURE.md`, `USAGE.md` 等、プロジェクトルート直下にあるもの）

※ `.kiro/` 配下、`node_modules/`、ビルド成果物ディレクトリは対象外です。
※ 対象ドキュメントが存在しない場合は「対象ドキュメントなし」と報告して検出結果は「該当なし」としてください。

【検出の観点】
仕様書の内容と既存ドキュメントを突き合わせ、以下のいずれかに該当する箇所を特定してください：
- 既存ドキュメントに記述されている機能・コマンド・API・設定項目・画面・データ構造等が、今回の仕様で**変更・削除・追加**される
- 既存ドキュメントの使い方・手順・前提条件が、今回の仕様の影響で**不正確になる**
- 既存ドキュメントのアーキテクチャ説明・ディレクトリ構成・技術スタック説明が、今回の仕様の影響で**実態と乖離する**

上記の影響がある場合、そのドキュメントを更新するタスクがtasks.mdに存在するか確認してください。tasks.md内を「README」「docs/」「ドキュメント」「document」などのキーワードで照合し、明示的に更新タスクが記述されているかを判定します。

【判定ルール】
- 影響あり × 更新タスクあり → 問題なし
- 影響あり × 更新タスクなし → **タスク漏れ**として報告
- 影響なし → 報告不要

（探索して見つかったドキュメントのパスを列挙。なければ「対象ドキュメントなし」）

**検出された影響とタスク漏れ**

| # | 影響を受けるドキュメント | 影響内容（仕様書側のどの変更が影響するか） | tasks.mdに更新タスクあり？ | 判定 |
|---|------------------------|----------------------------------------|---------------------------|------|
| 1 | （ファイルパス） | （仕様書の該当箇所と、ドキュメントのどの記述が影響を受けるか） | あり/なし | OK / タスク漏れ |

（検出されなければ「該当なし」）

**対応推奨アクション**
（「タスク漏れ」と判定した項目 → tasks.mdに該当ドキュメントの更新タスクを追加することを推奨）
